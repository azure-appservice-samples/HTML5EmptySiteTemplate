Objetivo

A aplicação desenvolvida tem como objetivo criar um aplicativo para reservas de salas que pode se comunicar com diferentes canais (ex.: Telegram,Skype, Slack, etc.). um canal de chat será criado e o usuário irá fazer uma solicitação de reserva de sala informando a data e o aplicativo retornara as salas disponíveis para aquela data exibindo algumas descrições importantes como capacidade e uma imagem ilustrativa. ao final da requisição o sistema irá gerar um protocolo e informar ao usuário sobre a reserva.

Arquitetura

  • SDK do Bot Framework Microsoft para C# : ultilizado para construção de uma aplicação REST que faz todo controle de fluxo de mensagem.
  • Microsoft Cognitive Services : Luis (Usado para identificar as intenções do usuário dentro do diálogo direcionando o mesmo para outros diálogos, sejá ele formulário ou FAQ)
  • Microsoft QnA Maker ultilizado para criar diálogos baseados em perguntas e respostas.
  • Microsoft Bot Connector ultilizado para connectar o meu bot nesta página ou em outros canais como skype.
  • Rest API : foi criado uma aplicação webApi no azure para consumo de dados.
  • Canal : Aproveitei a página default do meu bot para ultizar o Web Chat, por se tratar de uma interface mais simples onde não precisa adicionar o bot, ex:skype.

Fluxo de Comunicação

  • Tarefa : preencher um formulário de reserva de salas ou tirar algumas duvidas do usuário quanto as regras para a reserva.
  • Objetivo : descrito no tópico anterior.
  • Motivação do usuário : fazer uma reserva com mais agilidade a qualquer momento.
  • Passos :
    • 01 ao entrar no chat o usuário deverá informar sobre o interesse em fazer a reserva ou se deseja ter informações referentes as regras para a reserva.
    • 02 caso o usuário solicite fazer uma reserva será direcionado dentro do diálogo para um formulário e no final receberá um protocolo de reserva.
    • 03 caso o usuário solicite saber as informações será direcionado para um diálogo de FAQ.
  • Previsões : o aplicativo deverá ultilizar serviços de AI para interpretar as intenções de diálogo.

Documentação adicional

INTENÇÕES

Intenções representam o objetivo do usuário, ou seja, o propósito pelo qual ele enviou determinada mensagem. Por exemplo, a mensagem “Qual o saldo da minha conta corrente?” poderia representar a intenção consultar_saldo_conta_corrente. Para cada intenção devem ser fornecidos exemplos que representem aquela intenção, sendo que é recomendável um mínimo de 5 exemplos por intenção. Após isso, será capaz de treinar o modelo preditivo (a partir de algoritmos de aprendizagem de máquina) e com isso prever a intenção desejada pelo usuário. Em um post futuro eu explicarei com mais detalhes este processo de treinamento e classificação.

ENTIDADES

Entidades representam termos/objetos que são relevantes para o contexto da aplicação ou que fornecem uma clarificação para uma intenção. Por exemplo, caso houvesse uma intenção chamada consultar_saldo_conta, poderia ser necessário clarificar com o usuário se ele está se referindo à uma conta corrente ou conta poupança. Esta desambiguação pode ser feita utilizando entidades. As entidades são como modelos baseados em regra, no qual ela só será reconhecida caso o usuário tenha digitado exatamente o que foi cadastrado nela. Por exemplo, caso haja a entidade “conta corrente” e o usuário digite “conta corrent”, esta entrada não será reconhecida pela entidade. Apesar disso, as entidades são levadas em consideração no treinamento do modelo.

DIÁLOGOS

Os diálogos são responsáveis por guiar o usuário até a resposta final. Eles se baseiam em condições, que podem por exemplo verificar se uma certa intenção e/ou entidade foi reconhecida, e em caso positivo devolver uma resposta para o usuário ou disparar um fluxo, solicitando mais informações.
